# Store Product Images As Media Assets On Service API Local Storage First

Product display images and virtual try-on silhouettes are operator-maintained media assets that must be served reliably to admin and machine UIs. Product and variant records will reference platform media assets rather than storing arbitrary external URLs; V1 will store uploaded media on service-api managed local storage on the VPS and derive public asset URLs from those media records, because the current production footprint is a single low-volume VPS and the media asset boundary keeps a later COS, OSS, or S3 migration localized.

For the first product display image slice, `media_assets` is the platform media asset boundary. Product writes bind `products.display_image_media_asset_id`; the legacy `products.cover_image_url` is cleared and no longer used as source of truth. Local storage records keep `storage_provider=local`, `storage_key`, content type, byte size, SHA-256, original filename, and a generated public URL under `/api/media-assets/:id/content`. Deployments that need machine UIs outside the service-api origin should set `MEDIA_ASSET_PUBLIC_BASE_URL` so generated URLs are absolute and machine-reachable.

Upload and product binding require product write permission. Generated asset content URLs are public-read so machine runtimes can render sale views without admin credentials. Product display image uploads accept only JPEG, PNG, and WebP up to 5 MB; SVG and other image types are rejected. When object storage is introduced later, product/catalog contracts should continue to consume media asset ids and generated public URLs while only storage-provider resolution changes behind this boundary.
