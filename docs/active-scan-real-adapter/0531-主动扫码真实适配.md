# 0531 主动扫码真实适配调研

调研日期：2026-05-30

## 结论

当前代码不是只有“手动输入付款码”的测试界面。仓库已经有一条 `payment_code` 后端支付链路，也有 `vending-daemon` 读取串口文本扫码器、脱敏、去抖、发布扫码事件、把原始付款码提交给后端的骨架。

但它还不能认为已经完成了“机器自带 HID/扫码器扫用户付款码”的可靠生产适配。准确判断如下：

| 接入形态 | 当前适配状态 | 生产判断 |
| --- | --- | --- |
| 串口 / USB CDC / COM 口文本输出 | daemon 有 `serial_text` 读循环和 PTY 测试 | 有骨架，但仍有关键链路 bug 和可靠性缺口 |
| USB HID 键盘模拟输入，长串 + Enter | 只有枚举和配置项，没有真实键盘/HID 捕获实现 | 未生产适配 |
| 手动输入识别到的码 | 有 `/dev/payment-code-scan`，仅 mock daemon 可用 | 明确是开发/测试入口 |

最关键的问题是：真实扫码自动提交路径现在会把 source 写成 `"scanner"`，但后端共享 schema 只接受 `"tauri_scanner" | "browser_test" | "keyboard_hid" | "web_serial" | "manual_dev"`。因此 `vending-daemon` 串口读到码后，调用后端 `/machine-orders/:orderNo/payment-code/submit` 很可能会被 400 校验拒绝。这个问题足以说明当前还不是生产可用闭环。

## 代码证据

### 1. 后端付款码支付能力已经不是纯 mock

后端已经明确把付款码支付建成独立支付方式，而不是复用二维码主扫：

- `packages/shared/src/schemas/orders.ts` 定义 `payment_code` 的提交 schema，包含 `authCode`、`idempotencyKey`、`source` 和可选 `scannerHealth`。
- `apps/service-api/src/orders/orders.service.ts` 允许 `paymentMethod === "payment_code"` 搭配 `alipay` 或 `wechat_pay`，创建订单时不生成 `paymentUrl`，而是等待机器端提交付款码。
- `apps/service-api/src/orders/machine-orders.controller.ts` 暴露 `POST /machine-orders/:orderNo/payment-code/submit`。
- `apps/service-api/src/payments/payment-code-orchestrator.service.ts` 会创建 payment code attempt，调用 provider 扣款，处理成功、用户确认中、未知、失败，并触发查询/撤销。
- `apps/service-api/src/payments/alipay.provider.ts` 已实现 `alipay.trade.pay`，传 `auth_code` 和 `scene: "bar_code"`。
- `apps/service-api/src/payments/wechat-pay.provider.ts` 已实现微信 V2 `/pay/micropay`，传 `auth_code`。

所以后端支付域不是“只有手动测试页”。它已经朝真实付款码收单方向实现了不少核心逻辑。

### 2. daemon 有串口文本扫码骨架

`crates/vending-core/src/scanner.rs` 有通用分帧器：

- 支持 `crlf`、`lf`、`cr`、`none`。
- 丢弃控制字符。
- 付款码脱敏。
- 1.5 秒内同码去抖。

`apps/vending-daemon/src/scanner.rs` 有 `ScannerRuntime`：

- `serial_text` 会读取 `scannerSerialPortPath`。
- 用 `tokio_serial` 打开串口。
- 读到完整 frame 后，一路发送脱敏 `DaemonEvent::ScannerCode` 给 UI，一路把 raw `auth_code` 通过 channel 交给付款码 watcher。

`apps/vending-daemon/src/shutdown.rs` 会启动 scanner runtime 和 `run_payment_code_watcher`。watcher 收到 raw code 后，恢复当前订单并调用 `TransactionStateMachine.submit_payment_code()`。

`apps/vending-daemon/tests/scanner_vision.rs` 用 PTY 模拟扫码器输入，并断言事件里只有脱敏码，SQLite/日志不落付款码明文。

这说明当前串口文本扫码不是完全空白。

### 3. 手动扫码页是开发入口，不是生产入口

`apps/machine/src/views/PaymentCodeDevScanView.vue` 是“付款码模拟扫码”页面：

- 页面文案明确是 DEV。
- 只允许 `daemonClient.currentConnection?.mock === true` 时提交。
- 调用的是 daemon 的 `/v1/intents/dev-submit-payment-code`。

`apps/vending-daemon/src/ipc.rs` 的 dev endpoint 也限制 source 只能是 `manual_dev` 或 `browser_test`。

所以手动输入界面是开发/测试用，不应该被当成生产扫码适配。

### 4. USB HID 键盘输入没有真实生产实现

代码里确实出现了 `keyboard_hid`：

- `apps/vending-daemon/src/config.rs` 的 `ScannerAdapterKind`。
- `apps/machine/src/config/machine-config.ts` 和维护页配置项。
- `packages/shared/src/schemas/orders.ts` 的 `paymentCodeSourceSchema`。

但这些只是枚举/配置/来源值，不是实现。

具体缺口：

- `apps/vending-daemon/src/scanner.rs` 对 `KeyboardHid` 只是设置 source 为 `"keyboard_hid"`，没有 Windows Raw Input、全局键盘 hook、HID 设备读取或隐藏输入框逻辑。
- `ScannerRuntime::from_config()` 对非 `serial_text` 的 adapter 会退而使用 `serialPortPath` 或 `scannerSerialPortPath`，仍然走 `tokio_serial`。这不是键盘 HID 实现，还可能误把下位机串口当扫码串口打开。
- `apps/vending-daemon/Cargo.toml` 只有 `tokio-serial` 和 `windows-service`，没有 `hidapi`、Raw Input、keyboard hook 之类依赖。
- `apps/machine/src-tauri/src/scanner.rs` 看起来是旧的 Tauri 串口读码尝试，但 `src-tauri/src/lib.rs` 没有 `mod scanner`，也没有注册 `scanner_self_check` / `start_scanner` command；`src-tauri/Cargo.toml` 也没有 `tokio` / `tokio-serial` 依赖，因此这段并未进入当前 Tauri 生产链路。
- `apps/machine/src/daemon/migration-guards.spec.ts` 明确禁止生产 UI 再引用 `listenPaymentCodeScanned`、`startScanner`、`scannerSelfCheck` 这些旧 native scanner API。

结论：USB HID 键盘模拟输入还没有生产实现。

## 当前链路的关键风险

### 1. 自动扫码提交的 source 不合法

`apps/vending-daemon/src/shutdown.rs` 中，扫码 watcher 调用：

```rust
machine_state.submit_payment_code(&order_no, code, "scanner").await;
```

但 `packages/shared/src/schemas/orders.ts` 的 `paymentCodeSourceSchema` 不接受 `"scanner"`。后端 controller 使用 `paymentCodeSubmitSchema` 校验请求体，因此真实扫码自动提交会被校验拒绝。

这个问题比 UI 显示问题更底层：即使串口读码成功，也无法形成可靠扣款链路。

### 2. daemon 只按“当前订单号”提交，没有严格判断订单是否处于付款码等待态

`run_payment_code_watcher` 只检查当前 snapshot 是否有 `order_no`，没有本地判断：

- 当前订单是否 `paymentMethod === "payment_code"`。
- 是否仍在等待付款码。
- 是否已经支付成功、出货中或关闭。
- 扫码器是否应在当前页面/状态下启用。

后端会拒绝不匹配的订单，但 daemon 侧没有把这种拒绝变成明确 UI/健康状态。用户可能看到“已接收到付款码”，实际后端没有受理。

### 3. 失败后重扫的幂等 key 设计不支持自然重试

`LocalStateStore.get_or_create_payment_attempt_key()` 对同一个 order 复用同一个 `idempotencyKey`。后端 `PaymentCodeAttemptsService.createOrReplay()` 又会优先按同一个 key replay 旧 attempt。

这意味着第一次付款码失败后，用户刷新付款码再扫，daemon 仍可能带同一个 idempotency key，后端 replay 旧 attempt，而不是创建新 attempt。这和后端返回 `canRetry: true` 的语义冲突。

生产上应当按“每次有效扫码 attempt”生成新的幂等 key，只有网络重试同一次提交时复用同一个 key。

### 4. scanner 健康状态不足

当前 `/v1/scanner/status` 初始为 `online=false,message=unknown`，只有收到扫码事件后才更新为 online。打开串口失败、读失败、断开重连等状态没有持续写入 status cache 或 HealthAggregator。

`ScannerRuntime::run()` 如果打开串口失败会返回 Err，但任务只是 `tokio::spawn` 后放进 tasks；没有监控这个 Err 并更新健康状态。生产上会出现“扫码器已经不可用，但 UI/运维不知道根因”的情况。

### 5. 没有断线重连和端口枚举/自检

当前串口读循环打开失败直接结束，读失败也直接返回 Err。没有：

- 周期重连。
- COM 口枚举。
- 维护页扫码自检。
- 端口占用/权限错误的可读诊断。
- Windows COM 口热插拔处理。

### 6. UI/daemon 当前交易快照合同不一致

machine UI 的 `transactionSnapshotSchema` 期待完整字段：`orderId`、`paymentNo`、`paymentMethod`、`paymentStatus`、`paymentUrl`、`maskedAuthCode` 等。

但 daemon 的 `current_transaction` 和 `create_order_intent` 返回的是 `vending_core::domain::TransactionSnapshot`，只有 `orderNo`、`status`、`nextAction`、`updatedAt`。这会影响 payment 页面识别 `payment_code`、展示订单金额/支付状态、以及根据支付成功进入出货。

如果这条 UI/daemon 合同没有修正，扫码支付链路即使后端扣款成功，前端也难以可靠展示和路由。

### 7. Tauri native scanner 是遗留死代码

`apps/machine/src-tauri/src/scanner.rs` 里有串口读码、emit `payment-code-scanned` 的代码，但它没有被 `lib.rs` 注册。当前 Tauri `invoke_handler` 只有 `get_daemon_connection`。

同时 `apps/machine/src/native/scanner.ts` 的事件类型还包含 `authCode`，但 Rust 事件结构已经只包含 `maskedCode`。这说明旧 Tauri scanner 桥接已经不是当前主线，不应被误判为生产适配。

## 应放 Windows 服务层还是 Tauri UI 层

建议的生产边界：

```text
Windows Service / vending-daemon
  - 串口 / USB CDC / RS232 / TTL 扫码器读取
  - 分帧、去抖、脱敏
  - 每次扫码 attempt 幂等 key 管理
  - 付款码提交、查单、撤销、失败重扫
  - 本地状态、健康状态、断线重连、日志脱敏

Tauri UI
  - 展示“请出示付款码”
  - 展示扫码器状态、脱敏码、扣款进度
  - 提供维护页自检/配置
  - 可保留 keyboard wedge 开发/应急 fallback，但不作为生产主链路
```

理由：

1. 付款码是支付凭证，不能依赖 WebView 焦点、页面是否打开、前端路由是否活着。
2. Windows 工控机目标是 UI 可重启、daemon 常驻；扫码和扣款属于交易关键路径，应随 daemon 生命周期运行。
3. 串口/USB CDC/COM 口设备天然适合由 service 打开并持续监管。
4. daemon 才能统一处理本地 SQLite、幂等、断网重试、健康上报、日志脱敏和故障恢复。
5. UI 层适合人机交互，不适合持有付款码明文或决定扣款状态机。

但要特别区分 USB HID 键盘 wedge：

- 如果扫码器只能模拟键盘输入，它的输入通常进入当前用户 session 的焦点窗口。Windows Service 在 Session 0 中不应依赖接收这种键盘事件。
- 因此生产采购/配置应优先要求扫码器支持 USB CDC/COM、RS232/TTL 串口，或至少支持可被后台进程稳定读取的专用 HID/POS 模式。
- 如果短期只能使用键盘 wedge，Tauri UI 可以做隐藏 input / 全局 key listener 的 PoC 或应急 fallback：只在付款码等待页启用，收到 Enter 后立即通过 IPC 交给 daemon，由 daemon 生成 attempt 并提交后端。但这仍然不是推荐的量产主链路。

换句话说：生产主链路应该在 Windows 服务层；键盘模拟输入只适合 UI 层做过渡捕获，且最终扣款状态机仍应落在 daemon。

## 建议改造清单

### P0：先修断链问题

1. 把 daemon 自动扫码提交 source 从 `"scanner"` 改成合法值：
   - 串口/CDC 建议用 `"tauri_scanner"` 或把共享 schema 增加更准确的 `"serial_text"`。
   - USB 键盘 wedge 用 `"keyboard_hid"`。
   - Web Serial 调试用 `"web_serial"`。
2. `BackendClient.submit_payment_code()` 带上 `scannerHealth`，至少包含 online、adapter、port、message。
3. 每次新的有效扫码生成新的 attempt idempotency key；同一次网络重试才复用 key。
4. daemon watcher 只在当前订单为 `payment_code` 且 `waiting_payment` 时消费扫码，否则忽略并发脱敏事件/提示。

### P1：补齐 daemon scanner 生产能力

1. `ScannerRuntime` 增加打开失败、读失败、断线、重连中的健康状态。
2. 串口读失败后指数退避重连，不要任务静默退出。
3. 增加 scanner self-check / list ports / test read IPC。
4. 维护页根据 adapter 显示对应字段；`keyboard_hid` 不应展示 baudRate/frameSuffix 作为主要配置。
5. 串口和下位机串口必须分离配置，避免 `KeyboardHid` 分支误用 `serialPortPath`。

### P2：修 UI/daemon 交易合同

1. daemon 的 create/current transaction 返回 `CurrentTransactionSnapshot` 完整字段，和 `apps/machine/src/daemon/schemas.ts` 对齐。
2. 付款码 submit 后把后端返回的 attempt 状态、masked auth code、operator hint 写回本地 session。
3. 支付成功后 daemon 应进入 dispensing nextAction，UI 只根据 daemon snapshot 路由。
4. 失败可重扫时 UI 展示“请刷新付款码后重试”，daemon 新建 attempt。

### P3：决定 USB HID 策略

建议采购和装机阶段明确要求扫码器至少支持一种服务层可稳定读取的模式：

- 首选：USB CDC/COM 或 RS232/TTL 串口文本输出。
- 可接受：厂商提供稳定 SDK 或 POS HID 读取方式，且能在后台进程中工作。
- 不推荐量产主链路：只能模拟键盘输入到焦点窗口的 keyboard wedge。

如果必须先支持 keyboard wedge：

1. 在 Tauri UI 的付款码等待页实现隐藏 input 捕获。
2. 做焦点守护：进入支付页自动 focus，blur 后立即拉回。
3. 只接受长度/字符集合理、以 Enter 结束的输入。
4. 捕获后立刻清空输入，不落 localStorage，不写日志。
5. 通过 daemon IPC 提交，daemon 统一幂等和状态机。
6. 明确标记为 fallback，不作为服务层健康通过条件。

## 最终判断

当前代码已经超过“只有手动输入测试页”的阶段：后端付款码扣款、daemon 串口扫码骨架、脱敏/去抖测试都已经存在。

但针对采购扫码器的真实生产适配还没有完成。特别是 USB HID 键盘模拟输入没有实现；串口自动提交路径也存在 source 不合法、幂等重试、健康重连、状态合同不一致等阻断问题。

生产方向应继续沿 `vending-daemon` Windows Service 做主链路，优先把扫码器配置成串口/USB CDC/COM 模式由服务读取。Tauri UI 只负责展示和维护操作；键盘 wedge 捕获可以作为短期 PoC/fallback，但不应成为量产支付链路的唯一方案。
