# 0531 微信小程序支付调查

调查日期：2026-05-31

## 结论

个人身份可以尝试在微信公众平台注册小程序账号，但这不能解决当前微信支付联调问题。公开微信支付接入资料显示，小程序支付接入仍然需要微信支付商户号、已绑定的 AppID、商户 API 证书/密钥和支付产品权限；官方小程序接入指引列出的申请材料覆盖个体工商户、企业、党政机关/事业单位/社会组织等，没有个人主体材料。因此“个人小程序”不能直接作为售货机微信收款通道。

对当前售货机端两种支付方式的判断：

| 当前售货机支付方式                       | 微信对应产品                       | 是否可由个人小程序解决                                               | 是否必须引入新流程                                                                            |
| ---------------------------------------- | ---------------------------------- | -------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| 主动扫码：用户展示付款码，机器扫码器读取 | 付款码支付 / Micropay，V2 XML 接口 | 不能。仍需微信支付商户号、绑定 AppID 和付款码产品权限                | 不因小程序而新增流程；继续走 `payment_code + wechat_pay`，但需要真实商户权限才能联调          |
| 被动扫码：机器展示二维码，用户微信扫一扫 | Native 支付，V3 `code_url`         | 不能。Native 可使用已认证小程序 AppID，但前提仍是商户号与 AppID 绑定 | 若继续展示二维码给用户扫，可沿用 Native；若改成用户进入小程序内付款，则必须新增小程序支付流程 |

因此，小程序不是前两种能力的“沙箱替代品”。如果未来获得非个人主体小程序和微信支付商户号，现有被动扫码可以继续用 Native；主动扫码继续用付款码支付。只有当产品形态变成“用户扫码进入小程序，在小程序内完成购买和支付”时，才需要引入新的小程序支付链路。

## 官方资料要点

### 小程序支付需要商户能力

微信支付的小程序接入指引描述的支付产品是：用户打开商家小程序下单、确认支付、支付成功后返回小程序。申请材料列出的是个体工商户、企业、党政机关/事业单位/民办非企业/社会团体/基金会等主体材料，并说明交易按商户类目费率收取服务费。

来源：https://pay.wechatpay.cn/static/applyment_guide/applyment_detail_miniapp.shtml

微信支付 V3 文档把 JSAPI/小程序下单放在同一接口：`POST /v3/pay/transactions/jsapi`。请求体必须包含 `appid`、`mchid`、订单金额、回调地址等，并且 `appid` 需要与 `mchid` 有绑定关系。小程序侧调起支付则需要先拿到 `prepay_id`，再调用 `wx.requestPayment`。

来源：

- https://pay.wechatpay.cn/doc/v3/merchant/4012791856
- https://pay.wechatpay.cn/doc/v3/merchant/4012791898

### Native 二维码支付可以绑定小程序 AppID，但不是小程序支付

Native 支付下单接口返回 `code_url`，商户把它转成二维码展示给用户，用户用微信“扫一扫”进入支付确认页面。官方 Native 产品介绍写明其支持的应用类型包括已认证的小程序，但也要求主体类型是个体工商户、企业、事业单位、政府机关、社会组织等。

来源：

- https://pay.wechatpay.cn/doc/v3/merchant/4012791877
- https://pay.wechatpay.cn/doc/v3/merchant/4012791874

这说明：如果以后拿到已认证小程序和商户号，可以把小程序 AppID 作为 Native 支付的 `appid` 使用，并绑定到 `mchid`；但用户交互仍是“扫机器二维码付款”，不是进入小程序内 `wx.requestPayment`。

### 付款码支付仍是独立的被扫产品

微信付款码支付文档描述的场景是收银员或设备读取用户微信付款码，商户后台调用付款码接口扣款。接口入参包含 `appid`、`mch_id`、`out_trade_no`、金额和 `auth_code`；`appid` 必须与商户号绑定。官方还要求对 `USERPAYING`、系统错误等状态进行查单、轮询和撤销处理。

来源：https://pay.wechatpay.cn/doc/v2/merchant/4011937125

这与当前售货机主动扫码方向一致：扫码器读用户付款码，后端调用微信 V2 micropay，支付成功后再出货。小程序无法把这条链路变成“个人可联调”；它最多提供一个可绑定的 AppID，核心仍是商户号和付款码产品权限。

## 对当前系统的适配判断

### 1. 主动扫码：不应改成小程序支付

当前主动扫码本质是线下收银台的“商户扫用户付款码”。仓库现有设计已经把它抽象成 `payment_code + provider`，并区分微信、支付宝：

- 支付宝对应 `alipay.trade.pay`。
- 微信对应 V2 `/pay/micropay`。
- 售货机端只负责读取付款码、脱敏、提交给后端。
- 后端负责扣款、查单、撤销、幂等和最终支付状态。

这条链路不需要小程序页面，也不需要 `openid` 或 `wx.requestPayment`。如果强行改成小程序支付，交互会变成用户先打开/进入小程序，再在小程序内付款，机器扫码器就不再是核心支付入口。这会改变售货机的主动扫码设计，不是适配，而是新增产品流程。

建议：保留 `payment_code + wechat_pay`。微信真实联调阻塞点不是没有小程序，而是没有可用微信支付商户权限。

### 2. 被动扫码：可继续 Native，也可新增小程序内支付

当前被动扫码是“机器展示二维码，用户扫”。微信里对应两种可选路线：

| 路线           | 用户体验                                                               | 后端接口                                      | 机器端改动                                                           | 是否建议                                 |
| -------------- | ---------------------------------------------------------------------- | --------------------------------------------- | -------------------------------------------------------------------- | ---------------------------------------- |
| 继续 Native    | 机器屏幕展示微信支付二维码，用户微信扫一扫直接进入支付确认             | `/v3/pay/transactions/native` 返回 `code_url` | 现有 QR 支付链路基本可复用                                           | 适合当前售货机场景                       |
| 新增小程序支付 | 用户扫机器码进入小程序，下单或确认订单，在小程序内 `wx.requestPayment` | `/v3/pay/transactions/jsapi` 返回 `prepay_id` | 需要小程序前端、登录取 openid、订单绑定机器/商品、支付回跳和状态同步 | 只有需要会员、营销、复杂选购时才值得引入 |

如果目标只是让用户付款后机器出货，Native 更短：用户无需进入小程序业务页，不需要小程序登录态，也不需要维护小程序购物流程。

如果目标是把售货机扩展为微信生态入口，比如会员、优惠券、发票、订单列表、远程选品，小程序支付才有价值。但这会是第三条支付链路，而不是当前 QR 支付的参数替换。

## 如果引入小程序支付，需要新增的流程

小程序支付至少需要以下新增能力：

1. 微信小程序账号：非个人主体、完成认证、完成备案和类目审核。
2. 微信支付商户号：小程序 AppID 与 `mchid` 绑定，配置 API v3 密钥、证书/公钥、回调地址。
3. 小程序前端：用户扫码进入小程序，携带机器编号、货道或订单草稿参数。
4. 登录取身份：小程序 `wx.login` 后由后端换取该小程序 AppID 下的用户 `openid`。
5. 小程序下单：后端调用 `/v3/pay/transactions/jsapi`，传 `payer.openid`，拿到 `prepay_id`。
6. 小程序调起：前端调用 `wx.requestPayment`。
7. 后端确认：不能只信前端回调，仍以微信支付回调和主动查单为准。
8. 机器联动：支付成功后通过现有订单状态/MQTT/daemon 触发出货，处理超时、取消、重复支付和退款。
9. 交易类规范：若被平台认定为交易类小程序，还要按公众平台要求接入订单发货管理，否则可能被限制支付权限。

其中第 4 步和第 5 步是与 Native/付款码最大的差异：小程序支付需要 `openid`，并且必须由小程序运行环境调用 `wx.requestPayment`。

## 关于“没有微信沙箱”的联调策略

微信支付没有像支付宝沙箱那样完整覆盖普通商户真实链路的公开沙箱。没有商户号时，能做的只是：

- 保持 provider 层的 mock 和 contract tests。
- 对签名串、XML/JSON 构造、回调验签、状态机、幂等和脱敏做本地自动化测试。
- 用支付宝沙箱继续验证同构的主动扫码状态机。
- 等拿到微信商户号后，用小额真实交易验证 Native、付款码、查单、撤销和退款。

小程序开发者工具或个人小程序账号不能替代真实微信支付权限。若小程序支付权限被公众平台限制，`wx.requestPayment` 会失败，仍需登录公众平台查看权限通知。

## 建议路线

短期不建议为微信支付联调而引入个人小程序。应继续把微信支付适配分成两条真实商户链路：

1. `qr_code + wechat_pay`：保留 Native V3，展示 `code_url` 二维码。
2. `payment_code + wechat_pay`：保留 V2 付款码支付，补齐真实商户后的查单、撤销、双向证书和小额交易验收。

如果近期无法取得企业/个体工商户商户号，可以考虑服务商/聚合支付特约商户进件，但这仍然需要合规经营主体资料和微信支付审核，不是个人小程序能绕过的限制。

中期只有在业务确实需要微信内会员/营销/订单页时，再新增第三条 `mini_program + wechat_pay` 流程。新增时不要替换现有 Native/付款码抽象，而是在支付方法或入口层明确区分：

- `qr_code`：机器展示机构二维码。
- `payment_code`：机器扫描用户付款码。
- `mini_program`：用户进入小程序后由 `wx.requestPayment` 支付。
