# 真实支付、退款与恢复试点运行手册

## 范围

本运行手册适用于使用非沙箱支付通道的生产试点机器。内容覆盖二维码支付、付款码扫码、支付结果不确定时的对账、付款码撤销、退款申请与查询，以及通过操作员界面完成的人工处理。本流程不是受保护演练流程。

受保护支付演练模式保持独立。演练订单和测试记录不得作为生产支付验收依据，也不得调用真实支付通道的查询、撤销、扣款或退款路径。

## 就绪门禁

启用真实试点销售前，操作员必须检查：

- `GET /api/payments/ops/readiness`
- `GET /api/payments/ops/machines/:machineId/preflight`
- 管理后台：支付 -> 上线门禁

必需证据：

- `mock_provider_disabled` 在生产环境通过。
- `real_provider_config_present` 通过，且已启用支付宝或微信支付配置、商户身份、应用身份、加密凭据，以及必需的证书或公钥材料。
- `notify_url_static_check` 通过，支付通道回调路由和适合生产使用的 HTTPS 地址均有效。
- `machine_real_provider_options_available` 对试点机器通过。
- 当试点包含付款码扫码时，`payment_code.scanner_runtime.ready` 必须通过。若扫码器运行状态降级，只有二维码支付选项仍就绪时，二维码支付才可继续使用。

## 真实小额支付验证

在机器运行界面使用正常的支付方式选择流程。

- 二维码支付：选择二维码支付方式，完成一笔真实通道小额支付，并确认订单从等待支付进入履约流程。
- 付款码扫码：仅在扫码器运行状态就绪时选择付款码方式，出示真实小额顾客付款码，并确认订单进入履约流程。
- 扫码器降级或不可用：确认付款码方式已禁用或隐藏，同时已就绪的二维码支付方式仍可使用。

不得为了本验证添加扫码器绕过逻辑、直接调用支付通道、存储层修补，或仅开发人员可用的特殊订单状态。

## 操作员恢复入口

二维码支付结果不确定：

- 管理后台：支付 -> 支付流水 -> 手动对账。
- API：`POST /api/payments/:id/reconcile`，请求体为 `{ "reason": "..." }`。
- 该动作会向已配置的真实支付通道查询支付结果，并记录管理审计条目，内容包括操作者、原因、支付编号、通道状态，以及通道结果是否已应用。

付款码尝试结果不确定：

- 管理后台：支付 -> 付款码尝试 -> 查询或撤销。
- API：`POST /api/payments/payment-code-attempts/:id/query`，请求体为 `{ "reason": "..." }`。
- API：`POST /api/payments/payment-code-attempts/:id/reverse`，请求体为 `{ "reason": "..." }`。
- 响应和审计条目只能使用脱敏后的付款码数据。不得暴露原始授权码、扫码器原始文本、授权码哈希或已存储的支付通道原始报文。

退款申请与查询：

- 当实际出货结果需要退款时，通过订单恢复申请退款：调用 `POST /api/orders/:id/recovery-actions`，动作为 `request_refund`，并填写操作员备注。
- 通过管理后台查询退款状态：支付 -> 退款管理 -> 查询。
- API：`POST /api/payments/refunds/:id/query`，请求体为 `{ "reason": "..." }`。
- 该动作记录管理审计条目，内容包括操作者、原因，以及脱敏后的退款查询结果。

人工处理：

- 使用订单恢复和支付恢复动作，将事件标记或流转到人工处理。
- 每一次人工介入都必须包含操作员原因，便于后续在管理审计条目中复核。

## 需要保留的证据

- 支付运维就绪结果和机器预检结果。
- 小额二维码支付订单、支付编号、支付通道状态和履约流转记录。
- 小额付款码订单、付款码尝试状态、支付通道状态和履约流转记录。
- 扫码器降级证据，证明付款码不可用但二维码仍可使用。
- 二维码对账、付款码查询或撤销、退款查询，以及所有人工处理动作的管理审计条目。
- 所有退款验证对应的退款记录和退款对账尝试。

## 本地验证边界

自动化本地测试可以验证就绪契约、操作员 API 入口、审计原因传递、扫码器预检证据和付款码脱敏。没有真实支付通道凭据、真实顾客支付应用或付款码，以及生产试点机器时，本地测试无法完成真实支付或退款。
