# VEM 生产试点标准作业流程

本标准作业流程是单台生产试点机器的第一版操作流程，面向非开发操作员编写。机器面向顾客销售前必须完成每日检查。操作员每天都必须使用以下入口。第一版不包含外部通知服务。

只使用对顾客安全的标识。向顾客询问机器结果页或小票页显示的顾客订单凭证、可见下单时间，以及顾客预期购买的商品。不要要求顾客分享支付应用中的隐私信息，也不要要求顾客提供手机屏幕照片。

## 操作员入口

- 机器运行界面：面向顾客的购买屏幕。
- 机器维护界面：现场维护屏幕，用于查看就绪状态、库存、daemon 健康、扫码器状态、下位机状态，以及维护锁动作。
- 近场顾客喇叭现场验收运行手册：使用 [Near-Field Customer Speaker Field Acceptance Runbook](./near-field-customer-speaker-acceptance.md) 验收 Win10/Tauri 生产机器上的真实顾客音频区域。
- 管理后台：浏览器控制台，用于支付、订单、机器、退款、审计条目和生产就绪。使用 `/machines/:id` 查看机器心跳、货道销售状态、库存异常复核和机器阻断项。
- 库存对账入口：管理后台中的库存异常复核流程，动作为 `accept_machine_stock`、`reject_machine_stock`、`manual_correct`，以及可选的 `clearBlocker`。
- 当前版本没有控制台动作时，运营人员可使用的已发布 API：
  - `GET /api/payments/ops/readiness`
  - `GET /api/payments/ops/machines/:machineId/preflight`
  - `POST /api/payments/:id/reconcile`
  - `POST /api/payments/payment-code-attempts/:id/query`
  - `POST /api/payments/payment-code-attempts/:id/reverse`
  - `POST /api/payments/refunds/:id/query`
  - `POST /api/orders/:id/recovery-actions`
  - `POST /v1/stock/attestation`
  - `POST /v1/stock/movements`
  - `GET /v1/sale-readiness`
  - `POST /v1/maintenance/whole-machine-lock/clear`

每个恢复动作都必须记录操作员姓名、原因、相关时的顾客订单凭证，以及简短备注。不要在这些入口之外修改存储记录。

## 每日检查

每日检查是必需步骤。机器面向真实顾客开放前必须完成；任何可能影响销售就绪的事件之后也必须重复执行。

1. 在管理后台打开机器列表，确认试点机器在线、心跳较新，且没有活动的整机维护锁。
2. 打开支付 -> 上线门禁，并通过 `GET /api/payments/ops/readiness` 确认真实支付通道就绪。
3. 打开机器支付预检，确认 `GET /api/payments/ops/machines/:machineId/preflight` 对预期支付方式就绪。
4. 在机器维护界面确认 daemon 健康、销售就绪、扫码器状态、下位机状态、MQTT/后端连接和库存确认状态。
5. 如果维护屏幕不可用，使用 `GET /v1/sale-readiness`。
6. 在管理后台检查未解决的支付、退款、订单恢复和库存对账队列。开放销售前，必须通过库存异常复核流程处理每个库存异常复核案例。
7. 确认机器运行界面处于顾客自助机模式，且不暴露维护、调试、桌面或更新控件。
8. 在现场日志中记录检查结果、操作员、时间和所有阻断项。

如果就绪被阻断、支付方式不可用、库存确认缺失或过期、下位机离线，或整机维护锁处于活动状态，不要向顾客开放机器。

## 启动

1. 打开机器电源，等待机器运行界面出现。
2. 确认界面显示商品目录、离线或维护状态。不得向顾客显示 Windows 桌面。
3. 进入机器维护界面，确认 daemon 健康和 `GET /v1/sale-readiness`。
4. 如果销售就绪，返回机器运行界面并执行每日检查。
5. 如果机器启动后处于离线或锁定状态，先按离线与网络恢复或维护锁恢复处理，再面向顾客开放。

## 补货与盘点

首次装货、每日补货、商品替换和任何实物盘点修正均使用本流程。

1. 通过机器维护界面或管理后台将机器退出顾客服务。
2. 实地盘点每个活动货道，并将商品身份、货道编码、启用状态和数量与当前货道规划比对。
3. 首次生产装货或货道规划变更后，通过机器维护界面记录 `POST /v1/stock/attestation`。
4. 后续补货时，记录 `POST /v1/stock/movements`，流水类型为 `planned_refill`。
5. 盘点修正时，记录 `POST /v1/stock/movements`，流水类型为 `stock_count_correction`。
6. 确认 `GET /v1/sale-readiness` 显示物理库存确认已就绪。
7. 只有销售视图与实物商品一致后，才将机器恢复为顾客服务状态。

不要仅通过把商品放入机器来让货道可售。必须记录正式库存流水。

## 库存异常复核处理

当每日检查、`/machines/:id` 或库存对账入口显示由库存不一致导致的库存异常复核案例或货道阻断项时，使用本流程。

1. 复核完成前，保持受影响货道不对顾客服务。
2. 在管理后台打开机器 -> `/machines/:id`，复核库存异常复核案例、货道销售状态、机器上报数量、平台数量、关联订单或命令，以及阻断原因。
3. 选择动作前，实地检查受影响货道并盘点商品。
4. 只有信任机器上报库存时，才选择 `accept_machine_stock`。
5. 当不信任机器上报且当前平台数量应保持权威时，选择 `reject_machine_stock`。
6. 当实物盘点数量已知，且需要由操作员记录修正数量时，选择 `manual_correct`。
7. 只有库存和机械状态都确认可售后，才使用可选的 `clearBlocker`。
8. 在案例备注中记录操作员姓名、原因、实物盘点证据、动作，以及是否使用了 `clearBlocker`。

不要跳过库存异常复核案例，也不要在管理后台、库存对账入口或已发布操作员 API 之外修改存储记录。

## 支付恢复

二维码支付或支付结果不确定时，使用本流程。

1. 向顾客询问顾客订单凭证和下单时间。
2. 在管理后台打开支付 -> 支付记录，通过订单、支付编号或顾客订单凭证找到支付。
3. 使用支付 -> 手动对账，或调用 `POST /api/payments/:id/reconcile` 并填写原因。
4. 如果对账确认已支付但订单未履约，继续执行履约恢复。
5. 如果对账确认未支付或已关闭，向顾客说明结果；只有控制台显示没有待处理支付后，才请顾客重新购买。
6. 如果对账仍不确定，在管理后台将案例标记为人工处理，并记录审计备注。

原支付结果不确定时，绝不要要求顾客再次付款。

## 付款码恢复

付款码扫码结果未知、顾客确认中、超时，或与扫码器相关时，使用本流程。

1. 向顾客询问顾客订单凭证和下单时间。
2. 在管理后台打开支付 -> 付款码尝试。
3. 使用查询动作，或调用 `POST /api/payments/payment-code-attempts/:id/query` 并填写原因。
4. 如果该尝试不可支付或必须取消，使用撤销动作，或调用 `POST /api/payments/payment-code-attempts/:id/reverse` 并填写原因。
5. 如果支付已确认但商品未出货，继续执行履约恢复。
6. 如果扫码器降级，保持付款码不可用；只有预检显示二维码已就绪时，才使用二维码支付。

不要要求顾客暴露支付应用隐私信息。顾客订单凭证和操作员控制台记录足以用于客服查询。

## 退款处理

当操作员确认顾客已付款但未收到商品，且补偿出货不适合时，使用退款。

1. 使用顾客订单凭证在管理后台找到订单。
2. 先通过 `POST /api/orders/:id/recovery-actions` 记录实际出货结果。
3. 当结果需要退款时，使用动作 `request_refund` 并填写原因。
4. 打开支付 -> 退款并查询退款状态，或调用 `POST /api/payments/refunds/:id/query`。
5. 向顾客提供顾客订单凭证，以及管理后台中的退款状态说明。
6. 如果支付通道状态仍为待处理，在现场日志中记录跟进负责人和时间。

除非支付通道或管理后台明确显示，不要承诺准确的银行到账时间。

## 履约恢复

支付成功但出货、取货或商品交付不确定时，使用本流程。

1. 询问顾客订单凭证，并检查机器货道区域。
2. 在管理后台打开订单并找到该订单。
3. 从订单恢复面板使用 `POST /api/orders/:id/recovery-actions`。
4. 只有在顾客收到商品时，才选择 `confirm_dispensed`。
5. 当商品未离开机器或顾客无法取得商品时，选择 `confirm_not_dispensed`。
6. 确认未出货后，如果退款是正确的顾客补救方式，选择 `request_refund`。
7. 只有在库存和机器健康状态使操作员批准的替代出货安全时，才选择 `compensation_dispense`。
8. 在操作员备注中记录实物证据。

如果机器可能不安全或货道卡货，保持机器停止服务，并使用维护锁恢复。

## 维护锁恢复

机器被整机维护锁阻断时，使用本流程。

1. 在机器维护界面读取锁定原因和就绪阻断项。
2. 先修正实物原因：卡货、下位机故障、门未关、扫码器问题、网络问题或库存问题。
3. 修正后确认 daemon 健康和 `GET /v1/sale-readiness`。
4. 只能通过机器维护界面，或带操作员和原因调用 `POST /v1/maintenance/whole-machine-lock/clear` 来清除锁。
5. 恢复顾客服务前，重复执行每日检查。

原因未知时，不要为了继续销售而清除维护锁。

## 离线与网络恢复

VEM 生产试点只使用联网授权销售。第一版不允许离线销售、离线支付或离线出货。

1. 确认机器运行界面显示离线视图或已禁用销售。
2. 使用现场操作员可访问的机器和现场设备，检查本地电源、以太网/Wi-Fi、路由器和现场网络状态。
3. 在机器维护界面确认后端、MQTT、扫码器和下位机状态。
4. 在管理后台检查机器心跳和支付预检。
5. 网络恢复后，等待心跳和销售就绪状态变为就绪。
6. 重新开放销售前，执行每日检查。
7. 如果断网期间顾客处于活动交易中，基于顾客订单凭证执行支付恢复和履约恢复。

不要用现金、人工转账或离线承诺出货替代平台流程。

## 托管更新

托管机器更新是正常更新路径。

1. 安排维护窗口，并将机器退出顾客服务。
2. 确认没有活动顾客交易，也没有未解决的恢复动作。
3. 使用托管机器更新运行手册和已批准的清单/产物。
4. 独立更新 daemon 和机器 UI。UI 更新不得重启 daemon；daemon 更新不得结束机器 UI。
5. 将托管更新证据 JSON 与发布记录一起保存。
6. 确认 daemon 健康、UI 进程健康和 `GET /v1/sale-readiness`。
7. 重新开放销售前，重复执行每日检查。

如果回滚证据显示回滚失败，停止常规更新工作，并使用紧急远程访问。

## 紧急远程访问

紧急远程访问面向运营人员，不面向常规现场操作员。只有在管理后台、机器维护界面或托管机器更新无法恢复主机时才使用。

1. 确认运营负责人已批准本次会话，并记录原因。
2. 通过受控 Tailscale SSH 使用维护账号。不得使用自助机账号进行远程维护。
3. 收集证据：当前屏幕状态、daemon 健康、ready 文件状态、服务/任务状态、更新证据和相关日志。
4. 优先恢复托管更新备份或重启受影响服务/任务，而不是手工替换产物。
5. 主机恢复后，退出远程访问，重新执行每日检查，并在现场日志中记录动作。

紧急访问可以使用主机级命令，但支付、退款、履约、库存和维护锁的事件恢复仍必须通过本标准作业流程列出的操作员入口和 API 完成。

## 顾客投诉处理

1. 听取顾客问题，并询问顾客订单凭证、下单时间、商品和可见机器状态。
2. 不要询问支付应用密钥、私人账号信息或私人支付屏幕照片。
3. 使用顾客订单凭证在管理后台搜索。
4. 如果支付不确定，按支付恢复或付款码恢复处理。
5. 如果已支付但未收到商品，按履约恢复和退款处理处理。
6. 如果顾客已收到商品但仍有投诉，记录订单备注并升级给运营经理。
7. 向顾客提供案例状态和用于跟进的顾客订单凭证。
8. 记录机器是否可以继续开放。若不能，退出顾客服务，并按相关恢复章节处理。
