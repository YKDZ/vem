# 物理库存确认运行手册

生产试点销售前，必须先具备操作员确认的库存事实，机器才能被视为可面向真实顾客。VEM 不假设硬件能够统计每个货道里的每件商品。本步骤的事实来源是操作员确认，以及由 daemon 记录并被平台接受的正式机器库存流水事实。

## 初始确认

1. 确认平台当前启用的机器货道规划是预期的生产货道规划。
2. 将同一货道规划同步并应用到售货 daemon。
3. 将商品实际装入每个货道。
4. 通过机器维护界面记录 `POST /v1/stock/attestation`。
5. 对每个启用的货道规划货道，记录：
   - `slotId`
   - `slotCode`
   - `sku`
   - 实物 `quantity`
   - 该货道是否启用销售

daemon 会校验已确认的货道集合是否匹配当前启用的货道规划、SKU 和货道编码是否一致，以及数量是否超过容量。随后 daemon 会按货道记录 `stock_count_correction` 事实，并更新机器销售视图。

## 销售视图规则

启用且库存大于零的货道变为 `sale_ready`。

启用但库存为零的货道变为 `sold_out`，可售库存为零。

禁用货道变为 `frozen`，即使存在实物库存，可售库存也为零。

被 `needs_count`、`blocked_for_planogram_change`、`movement_rejected` 或 `needs_platform_review` 等对账状态阻断的货道，在通过正式盘点或补货流程解决前保持不可售。

## 后续库存变更

初始确认之后，操作员不得仅通过把商品放入机器来使库存可售。

后续变更使用 `POST /v1/stock/movements`：

- `planned_refill`：计划现场补货，数量为本次新增数量。
- `stock_count_correction`：观察盘点修正，数量为最终盘点数量。

每条流水都必须包含操作员归因。daemon 会存储本地流水，更新本地账本和销售视图；当机器配置包含机器编码和 API 基础地址时，还会将平台库存流水上传任务入队。

## 就绪证据

`GET /v1/sale-start-capability` 暴露面向顾客交易的当前阻断原因。库存确认尚未完成、过期或不一致时，销售视图不会给出 `sale_ready` 货道，因此该能力快照不会允许开始销售。

心跳中的 `statusPayload.physicalStockAttestation` 会把同一证据暴露给平台生产试点就绪评估器。

状态：

- `ready`：当前启用的货道规划已有确认记录，且当前本地库存状态一致。
- `missing`：尚未记录确认。
- `stale`：确认后当前启用的货道规划发生过变更。
- `inconsistent`：货道规划映射、货道启用状态或本地库存状态需要操作员复核。

对于生产硬件路径，缺失、过期或不一致的确认都会阻断联网授权销售就绪。

## 人工约束

本流程是操作员确认，不是自动货架感知。记录事实前，必须由人员确认实物商品身份和数量。

确认后若真实出货成功，daemon 会将本地库存扣减一次，并通过现有订单绑定库存流水路径记录面向平台的出货库存流水。
