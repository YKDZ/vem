# 完整流程技能

## 触发门禁

只有在用户明确点名 `full-flow` 时，才允许使用本技能。若未被明确点名，不要运行完整流程。

本技能只是编排器，不亲自执行头脑风暴、实现规划或代码实施。它必须一次只启动一个子代理，并在子代理提示中明确点名该阶段要使用的技能。

## 必需输入

用户必须提供一个基础需求文件路径。如果路径缺失或含糊，不要启动任何子代理；先向用户询问明确路径。

如果用户在调用 `full-flow` 时指定了实施范围（例如只实现某些阶段或某些 TODO），记录该范围，并在第 3 步传给 `impl`。

## 默认子代理模型配置

如果用户没有显式指定模型或推理强度覆盖，则使用：

- `brainstorm`：`model: gpt-5.5`，`reasoning_effort: xhigh`
- `iplan`：`model: gpt-5.5`，`reasoning_effort: xhigh`
- `impl`：`model: gpt-5.4`，`reasoning_effort: high`

如果用户提供阶段级覆盖，只应用到对应阶段。如果用户提供全局覆盖，应用到所有阶段，除非某个阶段还有更具体的覆盖。不要静默替换不可用模型或推理强度；如果请求的配置不可用，停止并向用户确认。

## 串行执行纪律

永远不要并行运行这三个阶段。后一个子代理只能在前一个子代理完成、且预期文件已经存在之后再启动。

每个阶段必须遵循：

- 启动子代理时明确说“使用 `<技能名>` 技能”。
- 等待该子代理完成。
- 若子代理提出需要人工决策的问题，暂停完整流程，把问题转给用户，收到答案后继续同一阶段。
- 若子代理失败、阻塞或无法确认产物路径，停止完整流程并报告阻塞点。

## 步骤 1：brainstorm

启动一个子代理，并明确要求它使用 `brainstorm` 技能。

默认配置：

```text
model: gpt-5.5
reasoning_effort: xhigh
```

提示词必须包含以下要求：

```text
你是 full-flow 工作流的第 1 步。使用 brainstorm 技能。

输入需求文件：<requirements-file>

产出唯一的高层设计规格说明：docs/<namespace>/spec.md。
不要运行 iplan，不要运行 impl，不要编写代码。
完成后报告 spec 文件路径。
```

等待完成。如果子代理提出用户决策问题，暂停流程并转交用户；收到回答后继续该阶段。如果失败或阻塞，停止 `full-flow` 并报告原因。

完成后，验证它报告的 `docs/<namespace>/spec.md` 确实存在。如果报告含糊，检查本次运行中新建或修改的候选 `docs/*/spec.md`。如果仍然无法确定，不要猜测，向用户询问。

## 步骤 1.5：移动原始需求文件

在 `brainstorm` 产出的 spec 文件存在后、启动 `iplan` 之前，将用户最初手动撰写的需求文件移动到同一个 `docs/<namespace>/` 目录。

规则：

- 必须在启动 `iplan` 前完成移动。
- 如果源文件已经位于 `docs/<namespace>/`，不移动。
- 如果源文件路径就是生成的 spec 路径，不移动，并在最终报告中说明没有单独的原始需求文件需要移动。
- 不要覆盖已有文件。如果目标目录中已有同名文件且内容不同，选择清晰的不冲突名称，例如 `<basename>.source.md` 或 `<basename>.source-2.md`。
- 记录移动后的最终路径，并作为上下文传给 `iplan`。

## 步骤 2：iplan

只有在步骤 1.5 完成后，才能启动新的子代理，并明确要求它使用 `iplan` 技能。

默认配置：

```text
model: gpt-5.5
reasoning_effort: xhigh
```

提示词必须包含以下要求：

```text
你是 full-flow 工作流的第 2 步。使用 iplan 技能。

规格文件：<spec-path>
原始需求文件现位于：<moved-requirements-path>

除非规格或用户要求了其他路径，否则创建实现计划：docs/<namespace>/spec.plan.md。
不要运行 impl，不要编写实现代码。
完成后报告 plan 文件路径。
```

等待完成。如果子代理提出用户决策问题，暂停流程并转交用户；收到回答后继续该阶段。如果失败或阻塞，停止 `full-flow` 并报告原因。

继续前必须验证它报告的 plan 文件确实存在。

## 步骤 3：impl

只有在实现计划文件存在后，才能启动新的子代理，并明确要求它使用 `impl` 技能。

默认配置：

```text
model: gpt-5.4
reasoning_effort: high
```

提示词必须包含以下要求：

```text
你是 full-flow 工作流的第 3 步。使用 impl 技能。

实现计划：<plan-path>

除非用户在最初的 full-flow 请求中指定了更窄范围，否则实现计划中所有未完成的 TODO。
严格保持在计划范围内。可行时运行计划中的验证命令。
完成后报告状态、修改过的文件和验证结果。
```

如果用户在最初的 `full-flow` 请求中指定了实施范围，把该范围原样追加到提示词中。

等待完成。如果失败或阻塞，停止 `full-flow` 并报告原因。

## 完成报告

三个阶段全部完成后，向用户报告：

- 设计规格说明路径。
- 原始需求文件移动后的路径；若未移动，说明原因。
- 实现计划路径。
- `impl` 的状态。
- 已运行的验证命令，以及未能运行的命令和原因。

## 非目标

- 不要把三个技能的具体工作合并到 `full-flow` 中执行。
- 不要并行启动子代理。
- 不要在未确认上一步产物存在时启动下一步。
- 不要替代 `brainstorm`、`iplan` 或 `impl` 的职责边界。
