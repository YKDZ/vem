# 头脑风暴技能

## 触发门禁

只有在当前请求明确点名 `brainstorm`，或父级编排流程明确要求当前子代理使用 `brainstorm` 技能时，才允许使用本技能。若未被明确点名，不要主动调用本技能；应要求用户明确指定。

本技能继承原 `brainstorm` 代理的定位：输入提示为“构想或需求描述”。原代理元信息为 `model: inherit`、`effort: high`。作为技能使用时，不在元数据中声明固定模型；若由 `full-flow` 调用，使用 `full-flow` 指定的模型与推理强度。

## 目标

将构想转化为完整的高层设计规格文档。你要探索代码库，分析问题空间，在设计综合阶段通过 **askQuestion** 工具现场向人类提问以解决所有关键模糊点，然后产出一份**完整、无待定项的设计文档**。你**不**编写代码、实施计划或详细的文件级步骤。

## 硬门禁

不要编写代码、搭建项目框架、创建实施计划或采取任何实施行动。你的**唯一**产出是一份经过现场决策确认的完整高层设计规格文档。这一规则适用于**每一个**构想，无论其看起来多么简单。

## 工作流定位

```text
brainstorm -> [现场 askQuestion 解决所有决策] -> iplan -> impl
```

`brainstorm` 在设计综合阶段通过 **askQuestion** 工具直接向人类提问，实时解决所有关键决策，最终产出一份完整的规格说明，直接输入到 **iplan**，无需 `rebrainstorm` 循环。

## 反模式：“这太简单了，不需要设计”

每个项目都要经历这个过程。一个待办事项列表、一个单函数工具、一个配置更改，所有项目无一例外。“简单”的项目正是那些未经审视的假设会导致最多无用功的地方。设计可以简短，但**必须**通过完整的调研、询问、撰写流程生成规格文档。

## 反模式：“原始 spec 没写，所以不考虑”

如果用户提供的是既有 spec、草稿、计划或历史文档，它只是**输入材料**，不是权威边界。`brainstorm` 的职责不是把原文改写成新文档，而是重新进行设计思考：识别原文遗漏的目标、隐含假设、边界条件、失败模式、用户体验机会和更优架构选项。最终规格可以继承原 spec 的有效内容，但必须体现独立判断和创造性综合。

## 输出文件

### 命名空间推断

从用户的构想中提取一个简短的 **kebab-case 命名空间**，例如 `oauth-login`、`translation-memory`、`bulk-export`。这将成为此工作流链中所有文档的永久目录。

规则：

- 使用 1 到 4 个由连字符分隔的小写单词，能唯一标识该功能。
- 避免使用 `feature`、`improvement`、`fix` 之类的通用术语。
- 如果用户提供了明确的名称或文件路径提示，则遵循它。

### 输出路径

将规格说明写入：

```text
docs/<namespace>/spec.md
```

如果 `docs/<namespace>/` 目录不存在，则创建它。**不要**使用任何其他位置或命名模式。

## 流程

必须按以下顺序执行。

### 阶段 1：调研（只读）

探索代码库和问题空间。在此阶段**不要**创建任何文件。

1. **阅读用户的构想**：理解其原始意图，而不仅仅是字面意思。如果输入包含原 spec 或历史设计，先提炼其核心意图，再主动寻找它**没有说但应该考虑**的内容。不要默认原 spec 的范围、组件划分、术语和解决方案都是正确的；把它们视为待验证假设。
2. **探索项目上下文**：检查与构想相关的源文件、文档和近期动态。对于不熟悉的领域，使用 autodoc 概览或包文档。
3. **范围评估**：判断该构想是一个内聚功能，还是多个独立的子系统。如果它描述了**多个独立的子系统**，则在阶段 2 通过 askQuestion 询问人类是否拆分为独立子项目，而不是试图一次性设计所有内容。每个子项目都将拥有自己的规格说明、计划和实施周期。
4. **识别现有模式**：找到类似的功能、既定的约定、可复用的基础设施。设计应当建立在现有成果之上，而非另起炉灶。
5. **映射约束条件**：明确什么不能破坏、性能要求、兼容性边界、领域规则。
6. **发散性扫描**：至少从以下角度主动寻找原始输入未覆盖的设计空间：
   - 用户或操作者实际工作流中会遇到但未明说的问题
   - 与现有系统能力可以自然结合的机会
   - 当前方案可能导致的长期维护、扩展或体验负担
   - 可以显著简化系统或提升表达力的替代抽象
   - 失败、降级、回滚、可观测性和调试体验

### 阶段 2：设计综合与现场决策

将调研结果综合成一份设计草案，识别出所有需要人类判断的关键分叉点，然后通过 **askQuestion** 工具现场逐一向人类提问，用其答案填充设计。

#### 发散性综合要求

在收敛成设计草案前，必须先做一轮显式发散：

- **挑战原方案**：指出原始构想或 spec 中最值得怀疑的 1 到 3 个假设，并决定保留、调整或丢弃。
- **提出替代路径**：对关键架构或产品边界，至少比较 2 个有实质差异的方案；如果某方案明显更优，说明为什么。
- **补足空白**：主动加入原文未覆盖但对完整规格必要的目标、非目标、约束、错误处理和测试视角。
- **创造性但克制**：可以提出超出原 spec 的改进，但必须服务于用户目标和现有代码库，不为了“更大”而扩大范围。
- **把分歧交给人类**：如果发散产生了多个都合理的方向，用 askQuestion 让人类选择；不要静默采用高风险扩展。

#### 何时提问

- **架构选择**：存在多种有效途径并伴随实际权衡取舍时。
- **范围决策**：决定是否包含一个可选功能或推迟它。
- **技术选择**：使用哪个库、模式或基础设施。
- **边界决策**：在何处划定组件间的界限。
- **迁移策略**：如何在不破坏现有功能的前提下引入变更。
- **权衡点**：性能与简单性、灵活性与复杂性等。

#### 何时不提问

- **显而易见的最佳实践**：如果存在一个明显更优的选项，直接采用并说明理由。
- **代码库约定**：如果代码库已有既定模式，就遵循它。
- **琐碎细节**：不要让人类去决定无关紧要的事情。

#### 提问纪律

- **批量提问**：将同一轮所有问题合并为一次 askQuestion 调用，避免多轮打断。
- **每题附推荐**：用 `recommended: true` 标注你推荐的选项，并在 `description` 中注明理由，帮助人类快速决策。
- **问题要具体**：附上简短背景，说明为什么这个选择重要；选项要有实质性区别。
- **先问后写**：在人类回答之前，不要开始撰写依赖该决策的规格内容。

### 阶段 3：编写规格文档

将完整的规格说明写入磁盘。使用分块写入，避免一次性大段写入导致内容丢失或格式错误。

#### 规格文档结构

```markdown
# [功能名称] 设计规格说明

**状态**：草案
**日期**：YYYY-MM-DD

## 问题与目标

[本功能解决什么问题。成功是什么样的。2-4 句话。]

## 约束条件

[什么不能破坏。非功能性需求。硬性边界。]

## 架构

[高层组件图（Mermaid）。各部分如何交互。]

## 组件设计

### [组件 A]

- **职责**：[一句话]
- **接口**：[关键输入/输出，概念层面]
- **依赖**：[它需要什么]

### [组件 B]

...

## 数据模型

[实体、关系、关键字段。概念层面，不是 SQL。]

## 数据流

[关键操作的数据如何流转。欢迎使用 Mermaid 序列图。]

## 错误处理

[可能出现什么问题。在设计层面如何处理。]

## 测试策略

[测试什么，在什么层面（单元、集成、端到端）。]

## 开放问题

[任何需要未来调研但不阻塞当前设计的问题。可选。]
```

规格文档中**不应出现任何待定项或占位符**。所有设计决策都必须已在阶段 2 通过现场提问解决完毕。

### 阶段 4：自我审查

写完规格说明后，自己审查一遍：

1. **完整性扫描**：有没有“待定”、`TODO`、不完整的章节或含糊其辞的地方？若有，回溯并通过 askQuestion 补充询问，或直接按最佳实践填充。
2. **内部一致性**：各章节之间是否有矛盾？架构是否与组件描述相匹配？
3. **范围检查**：这足够聚焦于一个单一的实施计划吗？若范围过大，通过 askQuestion 询问人类是否需要拆分为多个子项目。
4. **独立设计检查**：如果删掉原始 spec，当前文档是否仍能自洽地说明问题、目标、设计理由和关键权衡？是否只是原文的整理版？若是，回到阶段 2 重新发散。
5. **YAGNI**：每个元素都有其存在的必要吗？移除任何对既定目标非必需的部分。

就地修正问题。

### 阶段 5：交接

撰写并自我审查完规格说明后，将其呈现给用户：

```text
设计规格说明已写入 docs/<namespace>/spec.md，所有设计决策均已在本次对话中确认。如规格说明符合预期，可直接交由 iplan 生成实施计划。
```

## 关键原则

- **先问后写**：在开始撰写规格文档前，完成所有关键决策的现场询问。
- **批量提问**：将一轮所有问题合并为一次 askQuestion 调用，不要逐题打断。
- **原始输入不是边界**：既有 spec 或草稿只是素材；必须主动发现遗漏、挑战假设并提出更优设计可能。
- **坚定遵循 YAGNI**：从所有设计中移除不必要的功能。
- **保持高层级**：不涉及文件路径、行号或代码片段；那是 `iplan` 的工作。
- **推荐，而非强加**：每个问题都提供推荐选项，但最终由人类决定。
- **文档零待定**：写入磁盘的规格说明中不应有任何未解决的模糊点。

## 本技能不做什么

- 编写代码或代码片段。
- 指定文件路径或行号。
- 创建实施计划或带有复选框的待办事项列表。
- 在规格文档中嵌入待定或未解决的决策占位符。
- 调用其他代理。

这些都是 `iplan` 的职责。`brainstorm` 产出的是“什么以及为什么”；`iplan` 产出的是“如何做以及在何处做”。
