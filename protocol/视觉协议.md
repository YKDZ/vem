# 视觉协议

> 日期：2026-07-01

## 1. 握手

售货机连接后发送：

```json
{
  "protocol": "vem.vision.v1",
  "type": "vision.hello",
  "messageId": "hello-001",
  "timestamp": "2026-06-29T10:00:00.000Z",
  "payload": {
    "clientRole": "machine",
    "machineCode": "M001",
    "protocolVersion": 1,
    "capabilities": [
      "profile_push",
      "presence_status",
      "person_departed",
      "ambient_light"
    ]
  }
}
```

视觉方返回：

```json
{
  "protocol": "vem.vision.v1",
  "type": "vision.ready",
  "messageId": "ready-001",
  "timestamp": "2026-06-29T10:00:00.100Z",
  "payload": {
    "serverName": "vem-vision-python",
    "serverVersion": "0.2.0",
    "cameraReady": true,
    "modelReady": true,
    "capabilities": [
      "profile_push",
      "presence_status",
      "person_departed",
      "ambient_light"
    ]
  }
}
```

若 `modelReady=false`，视觉方会继续推送 `vision.error`，常见 code 为 `model_not_ready`。

## 2. 心跳

售货机可发送：

```json
{
  "protocol": "vem.vision.v1",
  "type": "vision.ping",
  "messageId": "ping-001",
  "timestamp": "2026-06-29T10:00:03.000Z",
  "payload": {}
}
```

视觉方返回 `vision.pong`，payload 为空对象。

## 3. 来人辅助状态：`vision.presence_status`

视觉方只有在客户端声明 `"presence_status"` capability 时才推送该消息。

示例：

```json
{
  "protocol": "vem.vision.v1",
  "type": "vision.presence_status",
  "messageId": "status-vision-event-xxx",
  "timestamp": "2026-06-29T10:00:03.500Z",
  "payload": {
    "eventId": "vision-event-xxx",
    "detectedAt": "2026-06-29T10:00:03.500Z",
    "state": "approach",
    "reason": "person_present_but_not_close",
    "personPresent": true,
    "occupancy": {
      "state": "single",
      "confidence": 0.82
    },
    "closeNow": false,
    "close": false,
    "closeTrigger": null,
    "proximity": {
      "present": true,
      "close": false,
      "closeNow": false,
      "personReady": true,
      "personPresent": true,
      "largestPersonRatio": 0.12,
      "facePresent": false,
      "bodyPresent": false,
      "method": "person_detector+face_area_ratio"
    },
    "ambientLight": {
      "level": "dim",
      "measuredAt": "2026-06-29T10:00:03.500Z",
      "source": "camera",
      "confidence": 0.82,
      "sample": {
        "lumaMean": 74.5
      }
    }
  }
}
```

已在视觉方代码中出现或可能出现的 `state` 包括：

| state      | 语义                                                        |
| ---------- | ----------------------------------------------------------- |
| `empty`    | 当前未检测到有效人员。                                      |
| `approach` | 检测到有人，但尚未满足靠近/触发画像条件。                   |
| `occupied` | occupancy gate 锁定，已为当前人员推过画像，等待离开后复位。 |
| `unusable` | 有人或采样过，但有效帧/置信度不足，暂不能形成画像。         |
| `waiting`  | mock/等待类状态。                                           |

售货机侧判断“是否有人”时应优先使用 `payload.personPresent`，不要依赖 `state === "present"`，因为视觉方没有使用 `present/absent` 这组 state。

`presence_status.payload.occupancy` 用于表达当前交互区域的单人/多人二值检测结果，不要求也不允许依赖具体人数：

| 字段         | 类型                                       | 说明                                                                                               |
| ------------ | ------------------------------------------ | -------------------------------------------------------------------------------------------------- |
| `state`      | `none` / `single` / `multiple` / `unknown` | `none` 表示无人；`single` 表示单人；`multiple` 表示多人；`unknown` 表示视觉方暂不能判断单人/多人。 |
| `confidence` | number                                     | 可选，0-1，仅用于诊断和调试，不作为业务强依赖。                                                    |

`personPresent` 继续作为来人判断的兼容字段；软件侧不得要求业务方自行从 `occupancy.state` 推导 `personPresent`。当 `occupancy.state=multiple` 时，软件侧仍应视为有人，但不得使用本轮个体画像做推荐或订单画像快照。

若客户端声明 `"ambient_light"`，视觉方应在 `presence_status.payload.ambientLight` 中附带亮度三态：

| 字段         | 类型                      | 说明                                          |
| ------------ | ------------------------- | --------------------------------------------- |
| `level`      | `bright` / `dim` / `dark` | 软件侧语音种类选择只依赖这个三态。            |
| `measuredAt` | ISO datetime              | 亮度采样时间，应接近本次 `presence_status`。  |
| `source`     | `camera`                  | 当前固定为摄像头估算。                        |
| `confidence` | number                    | 可选，0-1，用于联调或诊断，不作为业务强依赖。 |
| `sample`     | object                    | 可选，原始或归一化亮度诊断，例如 `lumaMean`。 |

亮度 level 的语义：

| level    | 语义                         |
| -------- | ---------------------------- |
| `bright` | 明亮，适合正常来人播报。     |
| `dim`    | 昏暗，适合更明确的提示播报。 |
| `dark`   | 黑夜/很暗，适合低打扰播报。  |

`ambientLight` 缺省表示视觉服务暂未实现或客户端未声明 `"ambient_light"`；软件侧必须回退到普通来人语音，不得阻断来人检测或售卖。

## 4. 画像结果：`vision.profile_result`

视觉方检测到用户靠近并形成可用画像后推送：

```json
{
  "protocol": "vem.vision.v1",
  "type": "vision.profile_result",
  "messageId": "result-vision-event-xxx",
  "timestamp": "2026-06-29T10:00:04.000Z",
  "payload": {
    "eventId": "vision-event-xxx",
    "detectedAt": "2026-06-29T10:00:03.900Z",
    "occupancy": {
      "state": "single",
      "confidence": 0.82
    },
    "profile": {
      "personPresent": true,
      "heightCm": 172,
      "shoulderWidthCm": 43,
      "ageRange": "adult",
      "gender": "unknown",
      "bodyType": "regular",
      "upperColor": "dark",
      "confidence": 0.86
    },
    "quality": {
      "overall": "fair",
      "warnings": [],
      "profileUsable": true,
      "sampleCount": 3,
      "validFrameCount": 2,
      "minValidFrames": 1,
      "targetSampleCount": 5,
      "samplingMode": "approach_buffer_immediate_close",
      "proximity": {
        "present": true,
        "close": true,
        "closeNow": true,
        "closeTrigger": "close_now",
        "personReady": true,
        "personPresent": true,
        "largestPersonRatio": 0.21,
        "method": "person_detector+face_area_ratio"
      }
    }
  }
}
```

本仓库协议层接受并保留视觉方当前实际发送的 profile 字段：

| 字段              | 类型        | 说明                                                |
| ----------------- | ----------- | --------------------------------------------------- |
| `personPresent`   | boolean     | 是否检测到有效用户。                                |
| `heightCm`        | number/null | 粗略身高。                                          |
| `shoulderWidthCm` | number/null | 粗略肩宽。                                          |
| `ageRange`        | string      | `child` / `teen` / `adult` / `senior` / `unknown`。 |
| `gender`          | string      | `male` / `female` / `unknown`。                     |
| `bodyType`        | string      | `slim` / `regular` / `strong` / `unknown`。         |
| `upperColor`      | string      | 上衣颜色类别。                                      |
| `confidence`      | number      | 0-1。                                               |

应用推荐层仍可只使用必要的非身份信号，例如身高、体型、颜色和置信度；但协议 schema 不应丢弃视觉方实际字段，否则维护诊断和联调会失真。

`profile_result.payload.occupancy` 必须是本次画像对应采样窗口的占用状态快照。它与最近一次 `presence_status.payload.occupancy` 语义一致，但软件侧处理画像时不需要回查上一条 presence 消息。

`quality.profileUsable` 是机器可读的画像可用性结论：

| 字段              | 类型                                                                                    | 说明                                                                              |
| ----------------- | --------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `profileUsable`   | boolean                                                                                 | `true` 表示可用于推荐；`false` 表示只能作为维护诊断，不得用于推荐或订单画像快照。 |
| `notUsableReason` | `multiple_people` / `no_person` / `low_confidence` / `insufficient_quality` / `unknown` | 可选，`profileUsable=false` 时建议提供。                                          |

当 `occupancy.state=multiple` 时，视觉方必须设置 `quality.profileUsable=false`、`quality.notUsableReason="multiple_people"`，并建议在 `quality.warnings` 中包含 `"multiple_people"`。多人场景下 `profile` 字段可以保留用于维护诊断，但软件侧不得将其当作有效个体画像。

## 5. 人离开事件：`vision.person_departed`

视觉方只有在客户端声明 `"person_departed"` capability 时才推送该消息。

该事件用于明确表达“上一位用户已经离开当前交互区域”。它不同于 `presence_status.personPresent=false` 的周期性无人状态：`vision.person_departed` 是边沿事件，应只在视觉方确认从有人/占用状态转为空场时推送一次，直到下一次有人后才允许再次推送。

示例：

```json
{
  "protocol": "vem.vision.v1",
  "type": "vision.person_departed",
  "messageId": "departure-vision-event-xxx",
  "timestamp": "2026-06-29T10:00:12.000Z",
  "payload": {
    "eventId": "vision-departure-xxx",
    "detectedAt": "2026-06-29T10:00:12.000Z",
    "lastSeenAt": "2026-06-29T10:00:10.800Z",
    "reason": "left_frame",
    "absenceDurationMs": 1200,
    "ambientLight": {
      "level": "bright",
      "measuredAt": "2026-06-29T10:00:12.000Z",
      "source": "camera",
      "confidence": 0.86
    }
  }
}
```

| 字段                | 类型                                                                                    | 说明                                                  |
| ------------------- | --------------------------------------------------------------------------------------- | ----------------------------------------------------- |
| `eventId`           | string                                                                                  | 本次离开事件 ID。                                     |
| `detectedAt`        | ISO datetime                                                                            | 视觉方确认离开的时间。                                |
| `lastSeenAt`        | ISO datetime/null                                                                       | 最近一次确认有人时间；未知时可为 `null` 或缺省。      |
| `reason`            | `no_person` / `left_frame` / `tracking_lost` / `absence_timeout` / `manual` / `unknown` | 离开判定原因。                                        |
| `absenceDurationMs` | number                                                                                  | 可选，从最近有人到确认离开的估算时长。                |
| `ambientLight`      | object                                                                                  | 可选，字段同 `presence_status.payload.ambientLight`。 |

视觉方生产要求：

- 当 occupancy gate 从 `occupied/approach/unusable` 等有人状态回到空场时，必须尽快推送 `vision.person_departed`。
- `person_departed` 不应等待画像推理，也不依赖完整 `profile_result`。
- 如果视觉方只能周期性推送 `presence_status.personPresent=false`，上位机仍会兼容处理；但生产协议要求额外提供 `person_departed` 边沿事件，以减少 UI 归位和后续空调控制延迟。
- 视觉方需要做去抖：短暂遮挡、单帧漏检不应立即触发离开；建议连续空场或 tracking lost 达到视觉侧阈值后再发。

## 6. 错误

视觉方异常时推送：

```json
{
  "protocol": "vem.vision.v1",
  "type": "vision.error",
  "messageId": "error-001",
  "timestamp": "2026-06-29T10:00:04.000Z",
  "payload": {
    "code": "camera_unavailable",
    "message": "camera unavailable",
    "retryable": true
  }
}
```

常见 code：

| code                  | 说明                                |
| --------------------- | ----------------------------------- |
| `invalid_message`     | JSON 或信封字段不符合协议。         |
| `unsupported_version` | `protocol` 不等于 `vem.vision.v1`。 |
| `camera_unavailable`  | 摄像头不可用。                      |
| `model_not_ready`     | 模型未就绪。                        |
| `internal_error`      | 视觉服务内部错误。                  |

## 7. 售货机侧实现原则

- WebSocket 连接后发送 `vision.hello`，capabilities 必须包含 `profile_push`、`presence_status`、`person_departed` 和 `ambient_light`。
- 收到 `vision.presence_status` 后，用 `payload.personPresent` 更新 UI 来人状态，并用 `payload.occupancy` 更新单人/多人状态。
- 收到 `vision.person_departed` 后，将当前 customer presence session 标记为离开，并在安全页面回到首页。
- 收到 `vision.presence_status.ambientLight.level` 后，将其作为来人语音种类选择输入；缺省时使用普通语音。
- 收到 `vision.profile_result` 后，先检查 `quality.profileUsable` 和 `payload.occupancy`；只有可用单人画像才进入推荐画像，并兼容更新来人状态。
- `quality.proximity`、`tracking`、`samples` 等字段保留为诊断数据；推荐算法可以忽略。
- 语音、推荐、UI 等前端功能实现者应优先消费机器端 Pinia store 暴露的稳定状态和 action，不直接解析 `latestDiagnosticPayload`。
- 没有 `presence_status` 或长时间无画像时，不阻断售卖，继续展示默认商品。
- 不再向视觉方发送 `vision.start_profile` / `vision.cancel`。

## 8. 视觉方低延迟保证

来人语音播报不得等待完整用户画像。视觉方必须保证：

- `presence_status` 是轻量来人状态事件，必须优先于或独立于 `profile_result` 推送。
- 当状态从无人变为有人，或检测到 `personPresent=true` 时，应立即推送 `presence_status`，并在同一 payload 内附带当前 `occupancy.state` 和 `ambientLight.level`。
- 亮度检测应使用当前帧或短滑窗，不能等待年龄、性别、身高、体型、颜色等画像推理完成。
- `profile_result` 可以稍后推送；软件侧语音播报只依赖 `presence_status.personPresent + presence_status.occupancy.state + presence_status.ambientLight.level`。
- 如果亮度暂时无法判断，视觉方仍应先推 `presence_status`，可以省略 `ambientLight` 或降低 `confidence`，不得因为亮度不可用而延迟来人状态。
- 无人状态下可以继续推送 `personPresent=false` 的 `presence_status` 与环境亮度，用于维护诊断；软件侧不得因此触发来人语音。
- 人离开事件必须作为独立轻量事件推送，不能等待下一次画像周期。

## 9. 生产化连接策略

本仓库的职责划分如下：

| 组件            | 职责                                                                                                     |
| --------------- | -------------------------------------------------------------------------------------------------------- |
| machine WebView | 持有长期 WebSocket 订阅；消费 `presence_status`、`person_departed` 和 `profile_result`；驱动 UI 与推荐。 |
| vending-daemon  | 只做视觉 readiness/self-check，不持有长期订阅，也不参与推荐计算。                                        |
| service-api     | 当前不直接消费视觉流；只通过机器端上报/诊断间接观察状态。                                                |
| vision-mock     | 本地联调模拟视觉端；必须按 hello capability 决定是否推送 `presence_status/person_departed`。             |

machine WebView 的长期订阅必须具备以下兜底：

- 周期发送 `vision.ping`，收到 `vision.pong` 后认为链路仍可用。
- pong 超时、WebSocket error 或 close 后，关闭旧 socket 并指数退避重连。
- 重连期间不阻断售卖；推荐回到默认商品。
- 本地 presence 状态必须有 TTL。若之前收到 `personPresent=true`，但后续长时间没有新的 `presence_status/profile_result`，UI 必须回到 unavailable/idle，不能长期显示有人。
- 来人判定使用竞速策略：视觉 `presence_status/profile_result` 和首次本地触屏/按键任一先到，都可以打开 customer presence session。
- 离开判定使用多级兜底：优先消费 `vision.person_departed`，兼容 `presence_status.personPresent=false/profile.personPresent=false`，最后用无操作超时将 session 标记为离开。
- 离开后只在商品浏览/确认等可安全放弃的页面自动回首页；支付、出货、结果、维护、离线、配网等流程不得被离开事件打断。
- 视觉侧发送 `vision.error` 时，软件侧记录诊断并回退默认推荐；错误是否可恢复以 `payload.retryable` 为参考，但不能让售卖主流程卡死。

## 10. 本仓库代码落点

- `packages/shared/src/schemas/vision.ts`：按视觉方实际消息支持 `vision.presence_status` 和 `vision.person_departed`，保留 profile 的 `shoulderWidthCm/ageRange/gender`，并支持 `occupancy.state`、`quality.profileUsable` 和 `ambientLight.level`。
- `apps/machine/src/native/vision.ts`：hello 声明 `presence_status/person_departed`，订阅回调处理 `onPresenceStatus/onPersonDeparted`，并负责 ping/pong、断线重连。
- `apps/machine/src/stores/vision.ts`：保存 `vision.presence_status/vision.person_departed` diagnostic，并通过 Pinia store 暴露稳定的 presence/occupancy/profile usability 投影。
- `apps/machine/src/composables/usePresenceInteraction.ts`：提供统一 customer presence session，支持视觉事件、首次本地交互、无操作离开兜底、UI 安全归位，以及按 `ambientLight.level` 选择来人音频变体。
- `crates/vending-core/src/vision.rs`：daemon readiness/self-check hello 声明相同 capabilities，但不做长期订阅。
- `apps/vision-mock/src/server.ts`：mock 协议支持 `presence_status/person_departed`，并按客户端 capability 推送 `presence_status/person_departed/ambientLight`，用于本仓库自动化测试。
